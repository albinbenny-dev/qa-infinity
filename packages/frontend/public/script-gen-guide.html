Script Generation Guide

End-to-end walkthrough for generating, reviewing, and manually correcting Robot Framework and Playwright scripts inside QA Infinity — including how to use the DOM Recorder tool to supply verified selectors.

OVERVIEW How script generation works

The script agent reads your test case (steps, expected result, use-case tag), any context hints you provide, and optional live DOM recordings. It then calls the AI model to produce a complete, runnable script. Robot Framework scripts go straight to GENERATED for manual QA review. Playwright scripts can optionally be live-tested and auto-healed.

What the agent uses as input

🤖 Always injected
  • Test case steps & expected result
  • Login flow from UI scanner
  • Navigation map & page locators
  • Shared .robot resource keywords
  • Recent auto-heals (to avoid repeating)
  • Golden script examples (few-shot)
📝 You supply (optional but powerful)
  • Context note — selector hints, routes, quirks
  • DOM Recording — live session capture with verified selectors
  • DOM Snippet — paste a single element from DevTools
  • Prerequisite TC — reuse login/nav from another script
  • Failed step info — when retrying after a run failure
💡
Best result formula DOM Recording + context note + prerequisite TC linked = the agent has real selectors, knows what's already working for login/nav, and won't invent locators.

PHASES Generation lifecycle

Every script job moves through these phases. Watch the queue panel in the Scripts page.

Phase Icon Meaning Your action
QUEUED ○ Waiting for a free worker slot. Wait.
GENERATING ⏳ AI model is generating the script. Wait (usually 15–40 s).
GENERATED ✓ Robot Framework only. Script written; ready for manual review. Open the script, review & run it manually.
QUEUED_VERIFY ○ Playwright only. Queued for live test run. Wait.
VERIFYING 🔬 Script is running against the live app. Wait.
HEALING 🩹 Auto-heal attempt in progress (up to 2 attempts). Wait.
VERIFIED ✓ Script passed live run. Done. Celebrate 🎉
MANUAL_REVIEW ⚠ Auto-heal exhausted or pattern unrecognised. Needs human fix. Read suspected issue → Retry with feedback or Regenerate.
FAILED ✕ Generation crashed (timeout, missing config, bad TC data). Check error message → fix data → Retry.
⚠
Robot Framework scripts are always GENERATED, never VERIFIED The auto-heal pipeline (verify → heal) is Playwright-only. For Robot Framework, you review and run scripts manually via RF CLI or the Execution tab.

STEP 1 Choose framework & mode

🤖 Robot Framework
  • Generates .robot files using RF Browser library (Playwright backend)
  • Human-readable keywords — easiest to read and correct manually
  • Injects shared .robot resource files (login, nav, common patterns)
  • Auto-Heal is disabled — you review and run manually
  • Default choice for the project
⚡ Playwright TS
  • Generates .spec.ts using @playwright/test
  • POM (Page Object Model) pattern — imports from ./pages/
  • Can use Generate + Heal mode: live-runs and auto-fixes up to 2×
  • Reading code requires TypeScript knowledge
  • Use when you need full Playwright feature set

Mode: Generate only vs Generate + Heal

ModeWhat happensWhen to use
⚡ Generate only Script is produced and saved. No live test is triggered. Always for Robot; Playwright when app is unavailable or you'll review first.
🩹 Generate + Heal Script runs immediately. If it fails, AI patches selectors and retries up to 2×. Playwright only — when you want a verified script with no manual effort.

STEP 2 Record with QA DOM Recorder

The DOM Recorder injects a floating panel into any live page and captures every click, fill, and navigation as structured selector data. Paste the export into the generation modal and the agent uses those selectors verbatim — no guessing.

How to inject the recorder

1
Open qa-dom-recorder.html in your browser (ask your team lead for the file location).
2
Option A — Bookmarklet: drag the "🎬 QA Recorder" button to your bookmarks bar. Click it on any page to inject.
Option B — Console: click "Copy Console Script", open DevTools (F12) in the target app, paste into the Console tab, press Enter.
3
A floating panel appears in the bottom-right corner. Click ▶ Start and perform your test scenario manually.
4
Click ⏹ Stop when done.
5
Click 📋 Export — the structured recording is copied to your clipboard.
6
Paste into the "DOM Recording" textarea in the QA Infinity generation modal.

What the recorder captures

FILL Same as CLICK + the value typed Input fields, search boxes, forms
NAVIGATE From/To URLs + sidebar HTML snapshot Page transitions, routing verification
TOAST_APPEARED Toast text, selector, duration (short-lived flagged) Success/error message assertions

The recorder ranks selectors best→worst and marks the top one as RECOMMENDED. The agent uses this as a locked locator — it will not substitute or invent alternatives.

💡
Toast tip If the recording captures a TOAST_APPEARED (SHORT-LIVED: Xms), the agent will add an immediate assertion right after the trigger step — no extra sleep needed.
⚠
Keep recordings under ~78 000 characters If the export is too large, the modal will show a warning. Record only the core happy-path steps.

STEP 3 Fill context & hints

Additional context textarea (optional but high-value)

Use this field to give the agent facts it can't derive from the test case alone:

  • The exact URL route: The primary sales route is /#/sales/primary-orders
  • A specific selector you know works: Submit button: input[type="submit"]
  • Post-login quirks: After login a project selector modal appears — dismiss it first
  • Two-step login: Enter username → click Login → enter password → click Login again
  • Env-specific notes: Use process.env.TEST_DEALER_CODE for the dealer code
✓
Save as default hints Check "Save as default hints for this test case" and the text is stored on the TC. Next time you generate or regenerate, it auto-populates the field — you don't retype it.

DOM Recording textarea

Paste the output from Step 2 here. When pasted, a green confirmation appears:

✓ Recording attached — agent will use captured selectors as locked locators.

Locked locators override the agent's default locator preference — it copies them verbatim.

STEP 4 Link a prerequisite TC

If your test case assumes a working login + navigation already covered by another TC's verified script, link that TC as a prerequisite. The agent imports the login/nav pattern directly from the existing script instead of generating it from scratch.

1
When generating a single TC, the prerequisite dropdown appears in the modal.
2
Search for the TC whose script handles the setup steps (e.g. "Login & Project Select").
3
Select it — a blue banner confirms: "Script will call [TC-001] setup before running its own steps."
💡
When to use If 10 TCs all start from the dashboard after login, link TC-001 (the login TC) as the prerequisite for all of them. The agent won't re-generate the login flow 10 times.

STEP 5 Submit & monitor queue

1
Select TCs in the left panel by clicking their checkboxes. Already-scripted TCs show ✓ and cannot be re-selected (use Regenerate instead).
2
Click ⚡ Generate (top bar) or 🩹 Generate + Heal (Playwright only). Fill the modal and click Submit.
3
Watch the queue panel on the right. Jobs appear in real-time. Hover the status icon to see heal attempt counts or error snippets.
4
When GENERATED or VERIFIED, click the ↗ button on the TC row to open the script in the Monaco editor. Review and run.

FIX Manual review state

A script enters ⚠ MANUAL_REVIEW when the auto-heal pipeline ran out of attempts or detected an issue pattern it couldn't resolve automatically.

Reading the suspected issue

The queue row shows a short suspected-issue message in amber (e.g. "Selector not found: css=#submit-btn"). This is the AI's diagnosis — use it as the starting point.

What typically causes MANUAL_REVIEW

  • Wrong selector — element exists but the generated locator doesn't match
  • Wrong route — the agent navigated to the wrong URL path
  • Timing — page takes longer to load than the default 10 s timeout
  • Multi-step login — login has an extra password confirm or MFA step
  • Post-login modal — a project selector or onboarding modal blocks navigation

FIX Retry with feedback

For jobs in MANUAL_REVIEW or FAILED state, click the ↺ Retry button on the queue row. This opens a feedback modal pre-filled with the last error — you describe what's wrong and the agent re-generates with that context.

Writing effective retry feedback

# Example — wrong selector
The submit button selector should be input[type="submit"], not #btn-login.

# Example — wrong route
The route after login is /#/FinanceUserListReport, not /finance/reports.

# Example — post-login modal
After login a project-selector modal appears. Click the first project card
before attempting to navigate to the dashboard.

# Example — two-step login
Login is two steps: enter username → click "Next" → enter password → click "Login".
The agent is combining both into one form which doesn't work.
💡
Combine retry with a DOM Recording Open the DOM Recorder, record the exact failing scenario manually, then paste the export into the Retry modal's DOM Recording field. The agent gets the exact selectors that work.

FIX Regenerate a script

Already have a script open in the editor and want to regenerate it? Click ↺ Regenerate in the toolbar above the editor. This opens the Regenerate modal with all fields pre-populated from the TC's stored hints.

Fields in the Regenerate modal

FieldPurpose
Issue description Free-text fix guidance (same as retry feedback). Pre-filled with stored hints.
DOM snippet Paste a single HTML element from DevTools — agent extracts the best locator.
DOM Recording Full recording export from QA DOM Recorder.
Failed step & error Auto-filled when you click "Fix with AI" from a run result. Shows the exact step + error.
Prerequisite TC Change or add a prerequisite link.
Script Framework Switch between Robot and Playwright (re-generates in the other framework).
Save as default hints Persists the issue description as the TC's stored hints for next time.

FIX DOM snippet helper

When you know exactly which element is failing but don't need a full recording, paste just the element's HTML from Chrome DevTools. The agent extracts the best locator using the priority chain: data-testid > id > aria-label > name > text > css.

1
Open the target app in Chrome. Right-click the problem element → Inspect.
2
In the Elements panel, right-click the highlighted node → Copy → Copy element.
3
Paste the HTML into the DOM snippet textarea in the Regenerate modal. 
<!-- Example paste -->
<button id="btnLogin" data-testid="login-submit" class="btn btn-primary">
  Sign In
</button>

The agent will use data-testid="login-submit" (highest priority) for this element.

ADVANCED Golden scripts

Mark a script as "golden" (⭐) by clicking the star icon on its TC row. Golden scripts are included as few-shot examples in the agent prompt for all future generation jobs in the same project.

  • Use for the cleanest, most representative scripts in your suite
  • The agent learns your project's coding style, keyword naming, and structure
  • 2–3 golden scripts is enough — don't mark everything
  • Click the star again to un-mark if a script goes stale
💡
Recommendation Mark one golden script per use-case tag (Primary Sales, Stock Management, etc.). This gives the agent diverse context for each domain.

ADVANCED Resource files

Shared .robot files live in scripts/{projectId}/resources/. Every Robot Framework generation job auto-injects all resource files into the script's Resource section, making their keywords available without you mentioning them.

Resource health banner

At the top of the Resources file tree, the banner shows total keyword count and script count. A green dot means resources are populated and healthy.

Keyword mining (⛏ Mine button)

Click the ⛏ Mine button to scan all existing .robot scripts for duplicate keyword patterns. The results suggest keyword blocks worth extracting into a shared resource file, reducing repetition across scripts.

💡
Upload a resource file Click ⬆ Upload Resource in the file tree. Resource files should contain only *** Keywords *** sections (no test cases). Login keywords, navigation keywords, and common assertion patterns are good candidates.

ADVANCED Import scripts

Use ⬆ Import (top bar) to bring in manually written scripts.

ModeWhat it does
Create TC from script QA Infinity reads the file, extracts the TC title and steps, and creates a new TC in DRAFT status automatically.
Link to existing TC Associates the uploaded file with a TC you pick from the library. Replaces any existing script for that TC.
Import standalone No TC link — custom script only. Shows up in the file tree but not tied to a TC.
✓
SeleniumLibrary auto-conversion Upload a .robot file that uses SeleniumLibrary keywords — QA Infinity automatically converts them to RF Browser (Playwright backend) equivalents before saving. A conversion confirmation banner appears after upload.

HELP Troubleshooting quick reference

SymptomLikely causeFix
Script generated but selector fails at runtime Agent guessed a CSS selector that's not unique or wrong Record with DOM Recorder → Regenerate with recording attached
Script navigates to the wrong URL Agent didn't know the correct route Add the correct route to context note: "Route is /#/primary-orders"
Login step fails (incorrect username/password field) Login form uses non-standard element structure DOM-record the login manually, paste recording, regenerate
Script stuck on project selector modal Post-login modal not handled Describe in context note: "A project selector modal appears after login — dismiss it by clicking the first project card"
Toast assertion fails ("element not found") Toast disappears before assertion runs DOM Recorder marks short-lived toasts — regenerate with recording; agent adds immediate assertion with no sleep
FAILED: "Script Agent is disabled" Script Agent toggled off in AI Usage settings Go to Project Settings → AI Usage → enable Script Agent
FAILED: "Test case or project missing" TC was deleted while the job was queued Re-create or restore the TC, then re-generate
Generated script has hardcoded URL Agent ignored BASE_URL rule (rare) Edit the script — replace the hardcoded URL with process.env.BASE_URL
Robot script uses SeleniumLibrary keywords An imported resource file still has old library references Use Import → the auto-converter will fix it, or update the resource file manually
Multiple TCs all regenerate with the same login pattern that's wrong Stored hints or a golden script contains bad login context Update the golden script first, or clear stored hints on affected TCs
🚨
Never hardcode credentials or project slugs in scripts The agent is instructed to use process.env.BASE_URL and process.env.TEST_PROJECT_SLUG. If you see hardcoded values, either edit them out manually or add an explicit instruction to the context note.
